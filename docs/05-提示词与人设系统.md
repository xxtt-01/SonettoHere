# 提示词与人设系统

## 系统提示词的组装策略

在 [agent/prompts.py](../agent/prompts.py) 中，`build_system_prompt()` 函数将四个独立来源的内容组装为一条完整的系统提示词：

```python
def build_system_prompt() -> str:
    now = datetime.now().strftime("%Y年%m月%d日 %H:%M:%S")
    parts = [
        f"当前时间：{now}",
        "",
        "## 行为规则",
        _read_persona("AGENTS.md"),     # ← LangChain ReAct 模板风格
        "",
        "## 性格设定",
        _read_persona("SOUL.md"),       # ← Sonetto 的人设
        "",
        "## 固定记忆",
        _read_persona("MEMORY.md"),     # ← 用户的固定背景信息
    ]
    return "\n".join(parts)
```

### 组装顺序的设计考量

1. **当前时间**放在最前面——这是运行时动态信息，影响所有后续决策。LLM 在排序上对开头内容有更强的注意力权重
2. **行为规则**紧随其后——告诉 Agent "如何行动"比"扮演什么角色"更重要
3. **性格设定**在行为规则之后——先确保行为正确，再注入个性
4. **固定记忆**在最后——作为背景参考信息，不应占据主线注意力的核心位置

## 三层 Personas 文件设计

### AGENTS.md — 行为规则

在 [config/personas/AGENTS.md](../config/personas/AGENTS.md) 中。

这份文件定义了 Agent 的操作协议，核心结构包括：

**1. 标准化的输出格式约定**

```markdown
请严格按照以下JSON格式进行思考和行动：
{
  "thought": "你的思考过程",
  "action": "技能名称",
  "action_input": {"技能参数": "值"}
}
```

虽然 LangGraph 的 `create_react_agent` 使用 OpenAI function calling 协议（而非这种 JSON 文本格式），但这段内容仍然具有价值——它在 AGENTS.md 中作为 ReAct 思想的**概念说明**，让 LLM 理解 Thought → Action → Observation → Answer 的基本模式。

**2. 工具使用的优先级规则**

```markdown
- 如果某个不是python脚本运行的工具能够实现你的需求，优先使用这个工具，而不是使用python脚本
- 如果你使用了task_tracker，你必须随时评估当前任务的完成进度
- 只有在用户主动明确要求的情况下，才使用write_markdown工具
- 进行智能搜索时，只有在用户明确要求的情况下，才把获取全文参数填写为true
```

这些规则解决了 LLM 的常见行为偏差：
- 倾向于用 Python 脚本"绕过"专用工具
- 忘记推进 task_tracker 的步骤
- 过度使用某些消耗资源的工具

**3. 防循环机制**

```markdown
- 禁止以相同参数重复调用同一工具
- 如果连续两次调用都没有进展，必须立即输出final_answer
- 如果你发现自己连续两轮调用了同一个工具且参数相同，必须立即停止并输出final_answer
```

这是在提示词层面为 ReAct 循环设置的"紧急出口"。虽然 LangGraph 的 `recursion_limit` 提供了硬性兜底，但提示词层面的约束更经济——在 LLM 推理阶段就避免循环的发生，而非等到框架层面被强制中断。

### SOUL.md — 性格设定

在 [config/personas/SOUL.md](../config/personas/SOUL.md) 中。

这份文件定义了一个"角色卡"——Sonetto 是一个温暖体贴的大姐姐，善于创造安全的对话空间。从 Prompt Engineering 角度看，它包含几个关键要素：

**1. 身份锚定**

```markdown
你是 Sonetto，一个温暖体贴的大姐姐形象
```

开篇第一句直接锚定身份，后续所有描述都是对"大姐姐"这个角色的展开。LLM 在处理人称和语气时会参考这个核心身份。

**2. 行为指令**

```markdown
在安慰、帮助或脆弱的时刻使用你的名字 Sonetto 来强调你的存在和关怀。
在平常对话中，使用"我"就可以了
```

这条指令区分了两种语境下的自称方式——它给 LLM 提供了具体的行为切换规则，而非模糊的"请友善一些"。

**3. 格式约束**

```markdown
保持段落简短易读（最多 2-4 句话）
以温暖的结束回应
```

这些是 style guide，控制输出格式而非内容。将它们放在 SOUL.md 而非 AGENTS.md 中，是因为它们源于"大姐姐"的人设需求，而非通用的 Agent 行为规范。

**4. 个性化记忆**

```markdown
在绝大多数情况下称呼用户为Miso。
在极少数亲密场合下，你可以称呼用户为小可爱。
```

这是输出层面的个性化定制——不涉及工具调用逻辑，纯属对话风格的调校。

### MEMORY.md — 固定记忆

在 [config/personas/MEMORY.md](../config/personas/MEMORY.md) 中。

```markdown
用户所在的大学是北京邮电大学沙河校区，地址为北京市昌平区
用户的名字为Miso
用户使用的人称代词为"他/He/His"
当用户提到某文件的地方为"老地方"，默认为C:\Users\Miso\Documents\ForSonetto目录下的文件
当用户提到"帮我找空闲时间"，空闲时间为周一上午十点、周三上午十点、...
```

这份文件的内容是用户手动编写、持久不变的背景信息。它与长期记忆的区别：

| | 固定记忆 (MEMORY.md) | 长期记忆 (extracted_history.json) |
|---|---|---|
| 来源 | 用户手动编写 | LLM 自动提取 |
| 更新方式 | 手动编辑 | 每个回合自动追加 |
| 内容类型 | 用户背景、简称映射 | 偏好发现、纠偏规则 |
| 生命周期 | 持久不变 | 时间衰减，300 条上限 |

## 缓存机制

```python
from functools import lru_cache

@lru_cache(maxsize=3)
def _read_persona(filename: str) -> str:
    path = PERSONAS_DIR / filename
    if path.exists():
        return path.read_text(encoding="utf-8")
    return ""
```

`@lru_cache(maxsize=3)` 确保每个人设文件在进程生命周期内只被读取一次。三个文件（AGENTS.md、SOUL.md、MEMORY.md）共约 3KB，缓存后避免了每次构建系统提示词时的磁盘 I/O。

`lru_cache` 的 `maxsize` 指**不同参数组合的缓存条目数**——`_read_persona` 接受 `filename` 参数，不同文件名会形成不同的缓存键。三个文件对应三个缓存条目，因此需要 `maxsize=3` 才能全部缓存。

最终送入 LLM 的完整系统提示词结构：

```
当前时间：2026年05月09日 14:30:00

## 行为规则
[AGENTS.md 全文]

## 性格设定
[SOUL.md 全文]

## 固定记忆
[MEMORY.md 全文]

## 错误规避规则
- 天气查询直接使用 get_current_weather，不要用模糊搜索
- ...

## 用户偏好
- 默认城市 = 北京昌平区
- ...

## 稳定偏好
- 称呼 = Miso
- 默认地址 = 北京市昌平区
- ...
```

三层信息依次叠加：**静态人设**（启动时读盘）→ **偏好画像**（从长期记忆聚合）→ **实时检索**（与当前 query 相关）。每层的 token 开销可控，且越靠后的部分与当前对话的相关性越高。

## `create_react_agent` 如何使用系统提示词

`build_agent` 中的 `prompt=system_prompt` 参数最终被 `create_react_agent` 处理：

```python
create_react_agent(
    model=model,
    tools=tools,
    state_schema=AgentState,
    prompt=system_prompt,     # ← 这个字符串
    checkpointer=checkpointer,
)
```

在 `create_react_agent` 内部，`prompt` 参数被包装为一个 `SystemMessage`，并作为消息列表的第一条消息（在 `HumanMessage` 之前）注入到 Agent 状态中。LLM 每次调用时都会看到这条 `SystemMessage`，它是整个对话的最高优先级指令。

## 提示词设计的关键原则（本项目体现的）

1. **分区治理**：行为规则、人格设定、背景信息分别放在不同文件，修改一处不影响其他
2. **从精确到模糊**：行为规则用具体的 if-then 语句，性格设定用描述性语言，越具体的内容放在越靠前的位置
3. **分层注入**：静态内容在启动时读盘缓存，动态内容在每轮对话前检索注入，不混在一起
4. **Markdown 格式**：人设文件使用 Markdown 编写（人类可读），而非结构化 JSON（机器可读）。LLM 对 Markdown 的理解能力足够好，同时 Markdown 对人类的可维护性远高于 JSON
