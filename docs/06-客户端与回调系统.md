# 客户端与回调系统

## 双入口架构

SonettoHere 通过 [main.py](../main.py) 的路由分发支持两种运行模式：

```python
mode = sys.argv[1] if len(sys.argv) > 1 else "cli"

if mode == "cli":
    from clients.cli import main as cli_main
    cli_main()
elif mode == "qqbot":
    from clients.qqbot import SonettoBot
    SonettoBot()
```

| 模式 | 入口 | 状态 |
|------|------|------|
| CLI | `clients/cli.py` → `SonettoCLI` | 完整实现 |
| QQ Bot | `clients/qqbot.py` → `SonettoBot` | 骨架（待 botpy SDK 集成） |

## CLI 客户端：异步事件流驱动

在 [clients/cli.py](../clients/cli.py) 中。

### 初始化流程

```python
class SonettoCLI:
    def __init__(self):
        settings = get_settings()
        self.session_id = datetime.now().strftime("%Y%m%d_%H%M%S") + "_" + uuid.uuid4().hex[:8]
        self.llm = ChatOpenAI(
            model="deepseek-v4-flash",
            api_key=settings.deepseek_api_key,
            base_url=settings.deepseek_base_url,
            temperature=0.7,
            streaming=True,                 # ← 启用流式输出
        )
        self.system_prompt = build_system_prompt()
        self.tools = get_all_skills()
        self.graph = build_agent(
            model=self.llm,
            tools=self.tools,
            system_prompt=self.system_prompt,
        )
        self.memory = ShortTermMemory()
```

初始化顺序反映了依赖关系：
1. 加载配置（`get_settings()`）
2. 生成会话 ID（时间戳 + 随机后缀）
3. 创建 LLM 实例（带 streaming）
4. 加载系统提示词（读盘 + 缓存）
5. 加载工具列表（30 个 Skill）
6. 构建 Agent 图
7. 初始化短期记忆

### 异步主循环

```python
async def run(self) -> None:
    while True:
        user_input = input(">>> ").strip()

        if user_input == "/exit":    break
        if user_input == "/clear":  # 重置记忆和回合消息
            self.memory.clear()
            self._turn_messages.clear()
            continue

        # 注入长期记忆 → 重新构建 Agent 图
        enhanced_prompt = self._build_enhanced_prompt(user_input)
        self.graph = build_agent(model=self.llm, tools=self.tools, system_prompt=enhanced_prompt)

        # 流式执行
        config = {"configurable": {"thread_id": self.session_id}}
        await self._run_stream_events(
            {"messages": [{"role": "user", "content": user_input}]},
            config,
        )

        # 回合结束 → 提取摘要 → 写入长期记忆
        self._save_turn()
```

### 每轮重建 Agent 图的原因

```python
self.graph = build_agent(
    model=self.llm,
    tools=self.tools,
    system_prompt=enhanced_prompt,  # ← 每轮的 prompt 不同（含检索结果）
)
```

这不是最优实现（更好的做法是只修改 `SystemMessage` 而非重建整个图），但它清晰地展示了概念：每轮对话的系统提示词是动态的，包含了基于当前用户输入的长期记忆检索结果。重建图的性能开销在 CLI 场景下可忽略不计。

### 流式事件处理

```python
async def _stream_events(self, inputs: dict, config: dict) -> None:
    async for event in self.graph.astream_events(inputs, config=config, version="v2"):
        kind = event["event"]
        name = event.get("name", "")

        if kind == "on_chat_model_start":
            # LLM 开始推理 → 青色 [Thinking] 头
        elif kind == "on_chat_model_stream":
            # LLM 逐 token 输出 → 实时打印思考内容
        elif kind == "on_chat_model_end":
            # LLM 推理完成 → 青色 [End Thinking]
        elif kind == "on_tool_start":
            # 工具开始执行 → 品红色 [Skill] + 黄色工具名
        elif kind == "on_tool_end":
            # 工具执行完成 → 绿色 [End Skill] + 耗时 + 结果
        elif kind == "on_tool_error":
            # 工具执行出错 → 红色错误信息
        elif kind == "on_chain_end" and name == "agent":
            # Agent 链结束 → 提取最终回答文本
```

`astream_events(version="v2")` 是 LangGraph 提供的流式事件 API。v2 版本区分了 `on_chat_model_stream`（LLM 流式输出）和 `on_chat_model_end`（LLM 调用完成），使终端能实时逐 token 打印思考过程，而非等整个推理完成后再一次性输出。

### 回合结束后的处理

```python
def _save_turn(self) -> None:
    extracted = extract_from_messages(self._turn_messages, self.llm)
    save_extracted(extracted, session_id=self.session_id)
```

每次 Agent 回答完成后，当前回合的用户消息、工具调用和最终回答被送入提取器，由 LLM 分析并写入长期记忆文件。

## PrinterCallback：独立的回调层

在 [callbacks/printer.py](../callbacks/printer.py) 中。

### LangChain Callback 系统

LangChain 提供了一套回调机制，允许在 LLM 调用、工具执行等关键节点插入自定义逻辑：

```
BaseCallbackHandler
    ├── on_llm_start()       ← LLM 开始推理
    ├── on_llm_new_token()   ← LLM 输出每个 token（流式）
    ├── on_llm_end()         ← LLM 推理完成
    ├── on_tool_start()      ← 工具开始执行
    ├── on_tool_end()        ← 工具执行完成
    └── on_tool_error()      ← 工具执行出错
```

### PrinterCallback 的实现

```python
class PrinterCallback(BaseCallbackHandler):
    def __init__(self):
        self._tool_start_time: dict[str, float] = {}   # 工具执行计时
        self._thinking_started = False

    def on_llm_start(self, ...):
        self._thinking_started = True
        print(f"{Fore.CYAN}┌── [Thinking]")

    def on_llm_new_token(self, token, ...):
        print(f"{Fore.CYAN}{token}", end="", flush=True)  # 逐 token 输出

    def on_tool_start(self, serialized, input_str, ...):
        tool_name = serialized.get("name", "unknown")
        run_id = str(kwargs.get("run_id", ""))
        self._tool_start_time[run_id] = time.time()        # 记录开始时间
        print(f"{Fore.MAGENTA}├── [Skill] {Fore.YELLOW}{tool_name}")

    def on_tool_end(self, output, ...):
        run_id = str(kwargs.get("run_id", ""))
        elapsed = time.time() - self._tool_start_time.pop(run_id)
        print(f"{Fore.GREEN}└── [End Skill]")
        print(f"{Fore.GREEN}    ├── 耗时: {Fore.CYAN}{elapsed:.2f}s")
```

### 彩色输出的设计

| 颜色 | 含义 | 使用场景 |
|------|------|----------|
| 青色 (Cyan) | 思考 | LLM 推理过程、Memory 信息 |
| 品红 (Magenta) | 行动 | Skill 调用入口 |
| 黄色 (Yellow) | 标识 | Skill 名称 |
| 绿色 (Green) | 结果 | Skill 执行结果、耗时 |
| 红色 (Red) | 错误/任务 | 工具错误、Task 跟踪 |
| 白色 (White) | 最终输出 | 最终回答 |

配色方案对应 ReAct 循环的四个阶段：思考（青）→ 行动（品红+黄）→ 观察（绿）→ 输出（白）。

### PrinterCallback vs CLI 内联输出

项目中存在两套输出方式：
1. **PrinterCallback**（`callbacks/printer.py`）—— 基于 LangChain Callback 机制的标准实现
2. **CLI 内联输出**（`clients/cli.py` 的 `_stream_events`）—— 基于 `astream_events` 的直接实现

CLI 实际使用的是第 2 种（`_stream_events` 方法内部直接调用 `print`）。`PrinterCallback` 作为独立的回调模块存在，展示了如何以符合 LangChain 规范的方式实现输出——在实际项目中，它可以被注册为全局回调：

```python
from callbacks.printer import PrinterCallback

# 在 ChatOpenAI 初始化时传入
llm = ChatOpenAI(
    model="deepseek-v4-flash",
    ...,
    callbacks=[PrinterCallback()],
)
```

两种实现分开的原因：`astream_events` 提供了更细粒度的事件控制（如区分 `on_chat_model_stream` 和 `on_chat_model_end`），而 `BaseCallbackHandler` 的回调机制更适合在非流式场景或作为全局钩子使用。

## 客户端设计的可扩展性

CLI 的 `SonettoCLI` 类可以很容易地抽象为通用接口，供不同客户端复用：

```python
class AgentClient:
    """伪代码：提取 CLI 中的通用逻辑"""
    def __init__(self):
        self.llm = ChatOpenAI(...)
        self.tools = get_all_skills()
        self.system_prompt = build_system_prompt()

    async def process_message(self, user_input: str, thread_id: str) -> str:
        enhanced = self._build_enhanced_prompt(user_input)
        graph = build_agent(self.llm, self.tools, enhanced)
        result = await graph.ainvoke(
            {"messages": [{"role": "user", "content": user_input}]},
            {"configurable": {"thread_id": thread_id}},
        )
        return result["messages"][-1].content
```

`SonettoCLI` 和未来的 `SonettoBot` 都可以基于这个通用接口构建，只需处理各自的输入输出通道。

## 配置层

在 [config/settings.py](../config/settings.py) 中。

### Pydantic BaseSettings

```python
class Settings(BaseSettings):
    deepseek_api_key: str = ""
    deepseek_base_url: str = "https://api.deepseek.com/v1"
    todoist_api_token: str = ""
    uapis_api_key: str = ""
    amap_api_key: str = ""
    qq_appid: str = ""
    qq_token: str = ""

    model_config = {
        "env_file": ".env",
        "env_file_encoding": "utf-8",
    }
```

`BaseSettings` 自动从 `.env` 文件和系统环境变量加载配置。字段名到环境变量名的映射是大小写不敏感的（`deepseek_api_key` 匹配 `DEEPSEEK_API_KEY`）。

### 单例模式

```python
_settings: Settings | None = None

def get_settings() -> Settings:
    global _settings
    if _settings is None:
        _settings = Settings()
    return _settings
```

整个进程只创建一次 `Settings` 实例，确保 `.env` 文件只读取一次，所有模块看到的是同一份配置。

### 为什么不用 `os.getenv` 散落各处？

- **类型安全**：Pydantic 提供类型校验（虽然这里都是 `str`）
- **集中管理**：新增 API Key 时只需在 `Settings` 加一个字段，无需全局搜索 `os.getenv`
- **默认值**：未配置的 Key 为空字符串，调用方可以检查并给出友好提示
- **文档性**：`Settings` 类的字段列表本身就是项目所需配置的完整清单
