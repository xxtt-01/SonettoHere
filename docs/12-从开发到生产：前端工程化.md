# 从开发到生产：前端工程化

## 开发模式 vs 生产模式

前端开发有两个截然不同的阶段，每个阶段有不同的需求和工具：

| 维度 | 开发模式 | 生产模式 |
|------|---------|---------|
| **目标** | 快速迭代、实时反馈 | 高性能、小体积 |
| **前端服务器** | Vite Dev Server（端口 5173） | FastAPI 直接 serve 静态文件 |
| **后端服务器** | Uvicorn（端口 8000） | 同一个 Uvicorn 进程（端口 8000） |
| **跨域问题** | 前后端不同端口 → 需要代理 | 同源 → 无需代理 |
| **文件体积** | 原始源码（未压缩） | 压缩、Tree-shaking、代码分割 |
| **错误提示** | 详细的 Source Map + DevTools | 生产错误最小化 |
| **热更新** | HMR（Hot Module Replacement） | 不适用 |

## Vite：下一代前端构建工具

### 为什么选 Vite？

传统构建工具（如 Webpack）在大型项目中启动可能需要几十秒。Vite 利用了浏览器原生 ES Module 支持，启动几乎是即时的：

```
Vite Dev Server:
  ┌──────────────────────────────────┐
  │ 源码文件 (.vue / .ts / .css)       │
  │   ↓ 按需编译（只在浏览器请求时）     │
  │ 编译后的 ES Module                 │
  │   ↓ 直接返回给浏览器                │
  │ 浏览器原生解析 import               │
  └──────────────────────────────────┘
```

对于本项目的 14 个源文件，Vite 开发服务器在毫秒级内启动。

### vite.config.ts 解析

```typescript
import { defineConfig } from 'vite'
import vue from '@vitejs/plugin-vue'
import { fileURLToPath, URL } from 'node:url'

export default defineConfig({
  plugins: [vue()],           // ① 支持 .vue 文件编译
  resolve: {
    alias: {
      '@': fileURLToPath(new URL('./src', import.meta.url)),  // ② 路径别名
    },
  },
  server: {
    port: 5173,               // ③ 开发服务器端口
    proxy: {
      '/api': 'http://localhost:8000',      // ④ REST API 代理
      '/ws': {
        target: 'ws://localhost:8000',       // ⑤ WebSocket 代理
        ws: true,
      },
    },
  },
})
```

#### ① Vue 插件

`@vitejs/plugin-vue` 让 Vite 能理解 `.vue` 文件——将它拆解为 template、script、style 三部分并分别编译。

#### ② 路径别名

`'@'` 映射到 `./src` 目录。所以在代码中可以写：

```typescript
import { useChat } from '@/composables/useChat'
// 等价于
import { useChat } from './composables/useChat'
```

别名的好处：无论文件嵌套多深，导入路径始终以 `@` 开头，不需要写 `../../../components/Foo.vue`。

#### ③ 开发服务器端口

Vite 默认端口 5173。如果需要修改，改这里的配置即可。

#### ④⑤ 开发代理（Proxy）

这是开发模式下最关键的配置。没有代理会有什么问题？

**问题：跨域（CORS）**

开发时，前端跑在 `localhost:5173`，后端跑在 `localhost:8000`。浏览器认为这是两个不同的"源"（origin = 协议 + 域名 + 端口）。从 `localhost:5173` 向 `localhost:8000` 发 `fetch` 请求会被浏览器的同源策略阻止。

虽然后端已经配置了 `CORSMiddleware`（`allow_origins=["*"]`），但 WebSocket 的跨域处理更复杂。代理是最干净的解决方案：

```
没有代理：
  浏览器 ──fetch /api/sessions──→ localhost:8000  ← 跨域，需要 CORS
  浏览器 ──ws /ws/chat──→ localhost:8000          ← WebSocket 跨域

有代理：
  浏览器 ──fetch /api/sessions──→ localhost:5173  ← 同源！
     Vite 内部转发 ──→ localhost:8000
  浏览器 ──ws /ws/chat──→ localhost:5173          ← 同源！
     Vite 内部转发 ──→ localhost:8000
```

从浏览器的视角看，所有请求都发给了 `localhost:5173`——前后端在同一个"源"上。Vite 在后台透明地将 `/api` 请求转发给 `localhost:8000`，将 `/ws` 的 WebSocket 连接也转发给 `localhost:8000`。

## 前端构建流程

### package.json scripts

```json
{
  "scripts": {
    "dev": "vite",
    "build": "vue-tsc --noEmit && vite build",
    "preview": "vite preview"
  }
}
```

### dev：启动开发服务器

```bash
cd web && npm run dev
```

Vite 在 `localhost:5173` 启动开发服务器，支持 HMR（修改代码后浏览器自动更新，无需刷新）。

### build：生产构建

```bash
cd web && npm run build
```

两步过程：

1. **`vue-tsc --noEmit`**：TypeScript 类型检查。`--noEmit` 表示只检查类型，不输出文件。如果类型错误，构建失败——这是一道质量防线。

2. **`vite build`**：Vite 将源码编译打包。输出到 `web/dist/` 目录：

```
web/dist/
├── index.html                          # 入口 HTML
└── assets/
    ├── index-C6_3gO-Z.js              # 所有 JS 打包为一个文件（含 Vue、vue-router、marked）
    └── index-D4TrykQp.css             # 所有 CSS 提取为一个文件
```

文件名中的哈希（`C6_3gO-Z`）来自文件内容的 MD5。内容变了，哈希就变了——这实现了"永久缓存"：浏览器可以永久缓存旧版本，新版本会自动获取新文件。

### 构建产物有多大？

对于本项目的 14 个源文件 + Vue 运行时 + vue-router + marked，生产构建后的 JS 文件通常在 100-150KB（gzip 后约 40-50KB）。这对于现代 Web 应用来说体积非常小——一个页面瞬间加载完毕。

## 生产模式：FastAPI 如何服务前端

### 服务静态文件

[`api/server.py`](../api/server.py) 的关键代码：

```python
from pathlib import Path
from fastapi.staticfiles import StaticFiles

WEB_DIR = Path(__file__).resolve().parent.parent / "web" / "dist"

def create_app() -> FastAPI:
    app = FastAPI(...)

    # ... REST API 路由、WebSocket 路由 ...

    # 生产模式：如果 dist 目录存在，serve 前端静态文件
    if WEB_DIR.exists():
        app.mount("/", StaticFiles(directory=str(WEB_DIR), html=True), name="static")
```

### 关键：`html=True` 的作用

`StaticFiles(html=True)` 做了两件事：

1. 当请求 `/` 时，自动返回 `web/dist/index.html`
2. 当请求的路径在目录中找不到时（如 `/memory`），回退到 `index.html`

第二点至关重要——因为 Vue Router 使用 `createWebHistory()`，前端路由的 URL（如 `/memory`）在服务器文件系统中并不存在。`html=True` 确保对这些"虚拟路径"的请求也返回 `index.html`，由 Vue Router 在客户端接管路由分发。

### 生产模式的请求处理

```
浏览器请求 http://localhost:8000/
    │
    ▼
FastAPI ← 匹配 / → StaticFiles → 返回 web/dist/index.html
    │
    ▼
浏览器解析 index.html，加载 /assets/index-*.js 和 /assets/index-*.css
    │
    ▼
Vue 应用启动，Vue Router 看到 URL 是 /
    │
    ▼
<router-view> 渲染 ChatView.vue
    │
    ▼
useSession 通过 fetch('/api/sessions') 恢复会话
useChat 通过 WebSocket('ws://当前origin/ws/chat/{id}') 连接
    │
    ▼
API 和 WebSocket 请求发给同一个 origin（localhost:8000），没有跨域问题
```

### 生产模式启动命令

```bash
# 先构建前端
cd web && npm run build

# 再启动后端（它会自动检测并serve dist/）
cd .. && python main.py web
```

后端自动检测 `web/dist/` 是否存在：如果存在，服务构建后的静态文件；如果不存在，仅提供 API 和 WebSocket 端点（纯后端模式）。

## CSS 主题变量系统

本项目没有使用 CSS 框架（如 Tailwind CSS），而是用 CSS 自定义属性（CSS Variables）构建了一套极简的暖色主题。

在 `App.vue` 的 `:root` 中定义：

```css
:root {
  --bg-primary: #faf8f5;       /* 主背景：暖白 */
  --bg-secondary: #f3efe8;     /* 次背景：浅米色 */
  --bg-card: #ffffff;          /* 卡片背景：纯白 */
  --text-primary: #3d342b;     /* 主文字：深棕色 */
  --text-secondary: #8b7e6e;   /* 次文字：灰棕色 */
  --accent: #b8956a;           /* 强调色：暖金色 */
  --accent-light: #dcc7a8;     /* 浅强调色 */
  --border: #e5ddd2;           /* 边框：浅米色 */
  --user-bubble: #e8dccf;      /* 用户气泡：偏暖 */
  --shadow: 0 1px 3px rgba(61, 52, 43, 0.06);  /* 阴影：极淡 */
  --radius: 10px;              /* 统一圆角 */
}
```

所有组件通过 `var(--xxx)` 引用这些变量：

```css
/* 组件中使用 */
.btn-send {
  background: var(--accent);
  color: #fff;
}

.session-item {
  border-radius: var(--radius);
}
```

CSS 变量的优势：
- **一键换肤**：只需改 `:root` 中的值，所有组件自动切换
- **设计一致性**：颜色、圆角等设计 token 全局统一
- **零运行时开销**：浏览器原生支持

## TypeScript 编译配置

[`web/tsconfig.json`](../web/tsconfig.json) 的关键配置：

```json
{
  "compilerOptions": {
    "target": "ES2020",
    "module": "ESNext",
    "moduleResolution": "bundler",
    "strict": true,
    "jsx": "preserve",
    "paths": {
      "@/*": ["./src/*"]
    }
  }
}
```

| 选项 | 含义 |
|------|------|
| `target: "ES2020"` | 编译到 ES2020（支持所有现代浏览器） |
| `module: "ESNext"` | 保持 ES Module 语法（Vite 负责最终打包） |
| `moduleResolution: "bundler"` | 告诉 TS 按打包工具的方式解析模块（理解 `@/` 别名） |
| `strict: true` | 启用所有严格类型检查 |
| `paths: {"@/*": ["./src/*"]}` | 让 TS 理解 `@/` 路径别名 |

## 没有使用的东西（以及为什么）

### 没有 Pinia / Vuex

状态管理用 composable 文件级单例替代。对于本项目只有两个页面、状态逻辑简单的场景，引入状态管理库是过度工程化。

### 没有 UI 组件库

从零手写每个 CSS 样式。这意味着你的 CSS 知识完全可移植——学会 `display: flex`、`border-radius`、CSS Variables 足以构建任何 UI。

### 没有 Tailwind CSS

Tailwind 是本项目的理想候选者，但为了教学目标，手写 CSS 让你看到每个样式规则的完整声明。理解了 CSS 原理后，再用 Tailwind 会更高效。

### 没有 axios

原生 `fetch` 够用且是每个前端工程师必须掌握的基础 API。

## 部署建议

当需要部署到服务器时：

### 方案一：一体化部署（推荐）

```bash
# 在服务器上
cd web && npm run build           # 构建前端
cd .. && python main.py web       # FastAPI 自动 serve dist/
```

只需暴露一个端口（8000），Nginx 反向代理即可。

### 方案二：分离部署

前端 `web/dist/` 部署到 CDN 或 Nginx，后端独立部署。需要在前端构建时配置正确的 API 地址。

### 无论哪种方案

- 生产环境确保 `allow_origins` 不设置为 `*`（CORS 安全）
- 使用 HTTPS + WSS（WebSocket Secure）
- 考虑在 Nginx 层配置 WebSocket 的 `Upgrade` 和 `Connection` 头
